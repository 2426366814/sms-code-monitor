# 多用户动态验证码监控系统 - 项目开发文档

## 1. 项目概述

多用户动态验证码监控系统是一个用于实时监控多个手机号验证码的网页应用，支持多用户管理、自动刷新、一键复制、导出数据等功能。

### 1.1 核心功能

| 功能 | 说明 |
|------|------|
| 自动刷新 | 支持1-10秒的刷新间隔 |
| 一键复制 | 支持复制手机号、验证码、原始信息 |
| 状态指示 | 显示有验证码、无验证码、请求错误等状态 |
| 数据导出 | 支持导出监控项数据为JSON格式 |
| 通知提示 | 收到新验证码时显示提示 |
| 多用户管理 | 支持用户注册、登录、权限管理 |
| API密钥管理 | 支持API密钥生成和管理 |
| 外部API接入 | 支持接入第三方接码平台 |
| 对外聚合API | 提供给外部应用调用的统一API |

### 1.2 技术栈

| 分类 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 前端 | HTML5/CSS3/JavaScript | ES6+ | 页面结构和交互 |
| 后端 | PHP | 7.4+ | 后端API开发 |
| 数据库 | MySQL | 8.0+ | 数据存储 |
| 认证 | JWT | - | API认证和授权 |
| 缓存 | Redis | 6.0+ | 缓存热点数据 |
| Web服务器 | Nginx/Apache | - | Web服务 |

#### 4.2.2 monitors 表（监控项表）

```sql
CREATE TABLE IF NOT EXISTS `monitors` (

  CONSTRAINT `monitors_ibfk_1` FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE
```

```sql
CREATE TABLE IF NOT EXISTS `codes` (
```

```sql
CREATE TABLE sms_platforms (
    id INT AUTO_INCREMENT PRIMARY KEY,

    api_key VARCHAR(255) NULL COMMENT 'API密钥',

    is_active TINYINT(1) DEFAULT 1,

    phone_field VARCHAR(50) DEFAULT 'phone',
    code_field VARCHAR(50) DEFAULT 'code',
    code_pattern VARCHAR(255) NULL COMMENT '验证码提取正则',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_platform_type (platform_type),
    INDEX idx_phone (phone)
```

```json
    "data": {},
```

### 5.4 对外聚合API（系统A）

**用途**: 将本系统所有验证码聚合，提供给外部应用调用

**路径**: `backend/api/external/v1/index.php`

**认证方式**: API Key

```
# 方式1: 使用Header（推荐）
X-API-Key: YOUR_API_KEY

# 方式2: 使用URL参数
?api_key=YOUR_API_KEY
```

#### API端点

```
基础路径: /backend/api/external/v1/index.php

GET  ?path=codes              获取验证码列表（支持分页）
GET  ?path=codes/{phone}      获取指定号码验证码
POST ?path=callback           接收外部推送验证码
GET  ?path=platforms          获取平台列表
GET  ?path=phones             获取可用号码
```

#### 获取验证码列表

```
GET /backend/api/external/v1/index.php?path=codes

可选参数:
- page: 页码（默认1）
- page_size: 每页数量（默认20，最大100）
- phone: 手机号筛选
- platform: 平台筛选
```

**响应示例**：
```json
{
    "success": true,
    "data": {
        "list": [...],
        "pagination": {
            "page": 1,
            "page_size": 20,
            "total": 100,
            "total_pages": 5
        }
    },
    "timestamp": "2026-03-01T12:00:00+00:00"
}
```

### 5.5 外部平台接入API（系统B）

**用途**: 接入第三方接码平台，从外部获取验证码

**路径**: `backend/api/external/platforms/index.php`

**认证方式**: JWT（用户登录后）

#### 平台类型说明

**类型1: 手机号码+网址形式**
- 一个手机号码对应一个API URL
- 每个手机号码需要单独请求
- 适用平台: 易码、接码平台等

**类型2: 直接网址形式**
- 一个API URL对应多个手机号码
- 一次请求返回所有手机号码的验证码
- 适用平台: SMS Pool、TextNow等

#### 添加外部平台

```
POST /api/external/platforms

类型1请求体:
{
    "name": "易码-13800138000",
    "platform_type": "type1",
    "phone": "13800138000",
    "api_url": "https://api.yima.com/sms?phone=13800138000&key=xxx",
    "auto_fetch": true,
    "fetch_interval": 60
}

类型2请求体:
{
    "name": "SMS Pool",
    "platform_type": "type2",
    "api_url": "https://api.smspool.net/messages",
    "api_key": "your-api-key",
    "response_type": "json",
    "phone_field": "number",
    "code_field": "code",
    "auto_fetch": true,
    "fetch_interval": 30
}
```

### 6.4 验证码提取规则

系统支持多种验证码格式：

| 格式 | 正则表达式 | 示例 |
|------|-----------|------|
| 中文格式 | `/验证码[是为：:\s]*(\d{4,8})/i` | 验证码是123456 |
| 英文格式 | `/code[：:\s]*(\d{4,8})/i` | code: 123456 |
| 手机关联 | `/{phone}.*?(\d{4,8})/` | 13800138000 123456 |
| 通用格式 | `/\b(\d{4,8})\b/` | 任意4-8位数字 |

## 16. 常见问题与解决方案

### 16.1 验证码弹窗重复问题

**问题描述**: 同一个验证码多次弹窗提醒

**根本原因**: 后端 `monitor_service.php` 每次刷新都更新 `code_timestamp`，导致前端无法判断是否是新验证码

**解决方案**: 只有当验证码**真正变化**时才更新 `code_timestamp`

```php
// backend/monitor_service.php
$oldCode = $monitor['last_extracted_code'] ?? '';

// 只有当验证码发生变化时才更新 code_timestamp
if ($extracted['status'] === 'success' && !empty($extracted['code']) && $extracted['code'] !== $oldCode) {
    $updateData['code_timestamp'] = $now * 1000;
    $updateData['code_time_str'] = $nowStr;
}
```

**相关文件**: `backend/monitor_service.php`

### 16.2 监控列表显示"暂无监控项"问题

**问题描述**: 监控列表有数据但仍显示"暂无监控项"

**根本原因**: `renderMonitors` 函数在添加监控行时，没有清除 HTML 中初始的 empty-state 行

**解决方案**: 在渲染监控列表前清除初始的 empty-state

```javascript
// index.html - renderMonitors 函数
// Clear any existing empty-state rows first
const emptyStateRows = tbody.querySelectorAll('tr:not([data-monitor-id])');
emptyStateRows.forEach(row => row.remove());
```

**相关文件**: `index.html`

### 16.3 手机号管理显示问题

**问题描述**: 手机号管理页面空白，不显示数据

**根本原因**: `loadPhones` 函数检查的数据格式错误，`data.data.length` 应为 `data.data.list.length`

**解决方案**: 修正数据格式检查

```javascript
// 修复前
if (data.success && data.data && data.data.length > 0)

// 修复后
if (data.success && data.data && data.data.list && data.data.list.length > 0)
```

**相关文件**: `index.html`

### 16.4 验证码文字不显示问题

**问题描述**: 监控面板和历史验证码页面的验证码文字不可见

**根本原因**: CSS样式中使用了 `background-clip: text` 和 `-webkit-text-fill-color: transparent`，导致文字透明

**解决方案**: 移除透明文字样式，添加正常的文字颜色

```css
/* 修复前 */
.code-display {
    background: var(--primary-gradient);
    -webkit-background-clip: text;
    -webkit-text-fill-color: transparent;
    background-clip: text;
}

/* 修复后 */
.code-display {
    color: var(--primary);
}
```

**相关文件**: `index.html` (`.code-display` 和 `.history-code` 样式)

### 16.5 验证码接收速度优化

**问题描述**: 验证码接收速度比直接访问API慢

**根本原因**: 
1. 后端守护进程刷新间隔过长（30秒）
2. 监控项之间有延迟（0.5秒）
3. 串行请求所有监控项

**解决方案**: 

1. **并行请求（curl_multi）**: 使用PHP的curl_multi同时请求所有监控项

```php
// backend/monitor_service.php
private function sendParallelRequests($urls) {
    $mh = curl_multi_init();
    $handles = [];
    $results = [];
    
    // 创建所有curl句柄
    foreach ($urls as $id => $url) {
        $ch = curl_init();
        curl_setopt_array($ch, [
            CURLOPT_URL => $url,
            CURLOPT_RETURNTRANSFER => true,
            CURLOPT_TIMEOUT => 10,
            CURLOPT_CONNECTTIMEOUT => 5,
            // ... 其他选项
        ]);
        curl_multi_add_handle($mh, $ch);
        $handles[$id] = $ch;
    }
    
    // 并行执行所有请求
    $running = null;
    do {
        curl_multi_exec($mh, $running);
        curl_multi_select($mh);
    } while ($running > 0);
    
    // 收集结果...
}
```

2. **守护进程间隔优化**: 从30秒减少到1秒

```php
// 修复前
public function runDaemon($interval = 30) { ... }

// 修复后
public function runDaemon($interval = 1) { ... }
```

3. **前端刷新间隔优化**: 默认从5秒减少到1秒

```html
<!-- index.html -->
<select id="refreshInterval">
    <option value="1" selected>1秒</option>
    <option value="2">2秒</option>
    <option value="3">3秒</option>
    <option value="5">5秒</option>
    <option value="10">10秒</option>
</select>
```

**性能对比**:

| 对比项 | 优化前 | 优化后 |
|--------|--------|--------|
| 刷新间隔 | 30秒 | 1秒 |
| 请求方式 | 串行 | 并行（curl_multi） |
| 26个请求耗时 | ~13秒 | ~500-700ms |
| 与单机版对比 | 慢6倍 | 相同 |

**相关文件**: `backend/monitor_service.php`, `index.html`
