# 多用户动态验证码监控系统 - 项目开发文档

## 1. 项目概述

多用户动态验证码监控系统是一个用于实时监控多个手机号验证码的网页应用，支持多用户管理、自动刷新、一键复制、导出数据等功能。

### 1.1 核心功能

| 功能 | 说明 |
|------|------|
| 实时监控 | WebSocket实时推送验证码（支持轮询回退） |
| 一键复制 | 支持复制手机号、验证码、原始信息 |
| 状态指示 | 显示有验证码、无验证码、请求错误等状态 |
| 数据导出 | 支持导出监控项数据为JSON格式 |
| 通知提示 | 收到新验证码时显示提示 |
| 多用户管理 | 支持用户注册、登录、权限管理 |
| API密钥管理 | 支持API密钥生成和管理 |
| 外部API接入 | 支持接入第三方接码平台 |
| 对外聚合API | 提供给外部应用调用的统一API |

### 1.2 技术栈

| 分类 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 前端 | HTML5/CSS3/JavaScript | ES6+ | 页面结构和交互 |
| 后端 | PHP | 7.4+ | 后端API开发 |
| 数据库 | MySQL | 8.0+ | 数据存储 |
| 认证 | JWT | - | API认证和授权 |
| 实时通信 | WebSocket | Node.js | 验证码实时推送 |
| Web服务器 | Nginx/Apache | - | Web服务 |

### 4.2 核心表结构

#### 4.2.1 users 表（用户表）

```sql
CREATE TABLE IF NOT EXISTS `users` (
  `id` INT(11) NOT NULL AUTO_INCREMENT,
  `username` VARCHAR(50) NOT NULL,
  `email` VARCHAR(100) NOT NULL,
  `password_hash` VARCHAR(255) NOT NULL,
  `is_admin` TINYINT(1) DEFAULT 0,
  `status` ENUM('active', 'inactive', 'suspended') NOT NULL DEFAULT 'active',
  `last_login` TIMESTAMP NULL DEFAULT NULL,
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `username` (`username`),
  UNIQUE KEY `email` (`email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

#### 4.2.2 monitors 表（监控项表）

```sql
CREATE TABLE IF NOT EXISTS `monitors` (
  `id` INT(11) NOT NULL AUTO_INCREMENT,
  `user_id` INT(11) NOT NULL,
  `phone` VARCHAR(20) NOT NULL,
  `url` VARCHAR(255) NOT NULL,
  `status` ENUM('loading', 'success', 'no-code', 'error') NOT NULL DEFAULT 'no-code',
  `last_code` VARCHAR(255) DEFAULT NULL,
  `last_extracted_code` VARCHAR(10) DEFAULT NULL,
  `code_timestamp` BIGINT(20) DEFAULT NULL,
  `code_time_str` VARCHAR(50) DEFAULT NULL,
  `last_update` VARCHAR(50) DEFAULT NULL,
  `last_update_timestamp` BIGINT(20) DEFAULT NULL,
  `message` VARCHAR(255) DEFAULT NULL,
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `user_id` (`user_id`),
  CONSTRAINT `monitors_ibfk_1` FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

#### 4.2.3 api_keys 表（API密钥表）

```sql
CREATE TABLE IF NOT EXISTS `api_keys` (
  `id` INT(11) NOT NULL AUTO_INCREMENT,
  `user_id` INT(11) NOT NULL,
  `name` VARCHAR(100) NOT NULL,
  `api_key` VARCHAR(191) NOT NULL,
  `api_secret` VARCHAR(255) NOT NULL,
  `status` ENUM('active', 'inactive') NOT NULL DEFAULT 'active',
  `expires_at` DATETIME DEFAULT NULL,
  `created_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  `last_used_at` DATETIME DEFAULT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `api_key` (`api_key`),
  KEY `user_id` (`user_id`),
  CONSTRAINT `api_keys_ibfk_1` FOREIGN KEY (`user_id`) REFERENCES `users` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

## 5. API接口文档

### 5.1 API基础规范

**请求格式**：
- 所有API请求使用JSON格式
- 请求头需包含 `Content-Type: application/json`
- 认证API需包含 `Authorization: Bearer {token}`

**响应格式**：
```json
{
    "success": true,
    "data": {},
    "message": "Success message",
    "code": 200
}
```

**基础URL**: `https://jm.91wz.org/backend/api/`

### 5.2 认证API

| 端点 | 方法 | 说明 |
|------|------|------|
| `/auth/index.php?action=login` | POST | 用户登录 |
| `/auth/index.php?action=register` | POST | 用户注册 |
| `/auth/index.php?path=me` | GET | 获取当前用户信息 |

### 5.3 监控项API

| 端点 | 方法 | 说明 |
|------|------|------|
| `/monitors/index.php` | GET | 获取监控列表 |
| `/monitors/index.php` | POST | 添加监控 |
| `/monitors/index.php?id={id}` | DELETE | 删除监控 |
| `/monitors/index.php?action=refresh&id={id}` | POST | 刷新单个监控 |

### 5.4 管理员API

| 端点 | 方法 | 说明 |
|------|------|------|
| `/admin/index.php?action=users` | GET | 获取用户列表 |
| `/admin/index.php?action=monitors` | GET | 获取所有监控项 |
| `/admin/index.php?action=stats` | GET | 获取系统统计 |
| `/admin/index.php?action=codes` | GET | 获取所有验证码记录 |

## 6. WebSocket服务

### 6.1 服务说明

WebSocket服务用于实时推送验证码更新，替代传统的轮询方式。

**服务位置**: `backend/websocket/server.js`

**默认端口**: 8080

### 6.2 启动服务

```bash
cd /home/wwwroot/jm.91wz.org/backend/websocket
npm install
npm start
```

### 6.3 使用PM2管理

```bash
pm2 start server.js --name websocket-server
pm2 status
pm2 logs websocket-server
```

### 6.4 连接方式

前端自动连接WebSocket，连接失败时自动回退到轮询模式。

```javascript
const wsUrl = `ws://jm.91wz.org:8080?token=${token}`;
const ws = new WebSocket(wsUrl);
```

### 6.5 消息格式

```json
{
    "type": "new_code",
    "data": {
        "phone": "13800138000",
        "code": "123456",
        "timestamp": "2026-03-03T12:00:00Z"
    }
}
```

## 9. 安全特性

### 9.1 权限隔离

- 管理员使用 `admin_token` 单独存储
- 普通用户登录时自动清除 `admin_token`
- 前端验证 `is_admin` 字段
- 后端API验证JWT中的用户角色

### 9.2 安全措施

| 措施 | 说明 |
|------|------|
| JWT认证 | 所有API请求需要有效Token |
| 密码加密 | 使用 `password_hash()` 加密存储 |
| SQL注入防护 | 使用PDO预处理语句 |
| XSS防护 | 输出转义和Content-Security-Policy |
| CORS配置 | 限制跨域请求来源 |

## 12. 常见问题

### 12.1 WebSocket连接失败

**现象**: 前端显示"WebSocket连接失败，使用轮询模式"

**解决方案**: 
1. 检查WebSocket服务是否运行: `pm2 status`
2. 检查端口8080是否开放
3. 检查防火墙配置

### 12.2 普通用户访问管理员页面

**现象**: 普通用户可以访问admin.html

**解决方案**: 已在v1.6.0修复，普通用户登录时会清除admin_token

### 12.3 验证码弹窗重复

**现象**: 同一个验证码多次弹窗提醒

**解决方案**: 已在v1.3.0修复，只有验证码变化时才更新code_timestamp
